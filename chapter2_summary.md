# Конспект главы 2. Проектная часть

> **Источник:** диплом, с. 42–68 (разделы 2.1–2.5).
> **Цель главы:** разработать проект автоматизации обработки банковских выписок в ООО «Социальные услуги».
> **Важное примечание:** в дипломе смешиваются два технологических подхода: доработка 1С:АБС и веб-приложение на PHP/MySQL. Ниже отражено содержание диплома с пометками о противоречиях.

## 2.1 Разработка проекта автоматизации

**Подход к разработке:**
- Использованы стандарты жизненного цикла: ГОСТ 34.601-90, ИСО 12207, ИСО 15288, RUP [диплом, с. 42–43].
- Выбрана методология **Rational Unified Process (RUP)** с четырьмя фазами: Введение, Дизайн, Строительство, Внедрение [диплом, с. 43–45].

**Этапы внедрения:** [диплом, с. 45–46]
1. Переосмысление и реорганизация существующего ПО.
2. Внедрение инженерами ИТ-отдела, поддержка специалистами 1С.
3. Тестирование бухгалтерами с поддержкой ИТ-отдела.
4. Создание документации по внедрению.

**Риски:** [диплом, с. 46–51, табл. 12]
- Неправильное определение масштаба и объёма проекта.
- Недостаточно ресурсов для тестирования.
- Недостаток опыта персонала заказчика.
- Увеличение нагрузки на персонал при внедрении.

**Меры снижения рисков:** чёткое планирование, привлечение внешних экспертов, обучение персонала, менеджмент качества.

> **[Рекомендация]** Добавить диаграмму Ганта и матрицу RACI для визуализации сроков и ответственности.

## 2.2 Управление проектом автоматизации

**Автоматизируемый процесс:** разнесение банковской выписки [диплом, с. 51].

**Типовой алгоритм:** [диплом, с. 51–57]
1. Получение выписки из интернет-банка или файлового архива (API, SFTP, веб-интерфейс).
2. Ввод и распознавание данных (парсинг, OCR для PDF).
3. Проверка корректности, фильтрация дублей, классификация операций.
4. Сопоставление реквизитов со справочниками (контрагенты, договоры, ЦФО, статьи ДДС).
5. Формирование бухгалтерских записей / платёжных поручений.
6. Передача данных в учётную систему (1С, SAP, Oracle и др.).
7. Контроль качества и обработка исключений.
8. Оповещение ответственных сотрудников.

**Моделирование:** приведены IDEF0-диаграммы, контекстная диаграмма, декомпозиция банковских операций, интерфейсы системы [диплом, с. 57–58, рис. 15–16].

## 2.4 Программное обеспечение задачи

**Архитектура:** веб-приложение с пользовательской и административной частями [диплом, с. 60–65].

> **[Технологический разрыв]** В главе 1 выбрано решение на базе 1С:Предприятие, а в главе 2 описывается веб-приложение на PHP/MySQL. В дипломе не объяснено, как эти компоненты связаны между собой и с 1С.

**Интерфейсы:**
- Интерфейс оплаты (выбор способа оплаты, получателя, суммы, подтверждение транзакции).
- Интерфейс администрирования (управление справочниками, пользователями, ролями) [диплом, с. 60–62].

**База данных:**
- Выбрана СУБД **MySQL** [диплом, с. 62].
- Описаны таблицы: пользователи, типы платежей, клиенты, БИК, способы оплаты.

> **[Обоснование выбора MySQL в проекте]** В качестве системы управления базами данных для разработанного веб-приложения выбрана СУБД MySQL. Это решение обусловлено рядом технических и экономических факторов, которые критичны для задачи автоматизации обработки банковских выписок в ООО «Социальные услуги».
>
> Во-первых, MySQL является традиционной и наиболее распространённой СУБД для веб-приложений, реализованных на языке PHP. Использование связки PHP + MySQL обеспечивает широкую совместимость компонентов, простоту развёртывания на типовых хостинговых и серверных платформах (LAMP/LEMP-стек), наличие большого количества готовых библиотек, драйверов и фреймворков (например, PDO, mysqli, Laravel, Yii). Это снижает трудозатраты на разработку и упрощает дальнейшее сопровождение системы.
>
> Во-вторых, MySQL полностью удовлетворяет требованиям по хранению структурированных данных, характерных для автоматизации бухгалтерских операций. В проекте необходимо хранить и оперативно обрабатывать справочники (контрагенты, банки, типы платежей, статьи движения денежных средств), транзакционные данные (платёжные операции, файлы выписок), а также служебную информацию (пользователи, роли, журналы действий). Реляционная модель MySQL позволяет формализовать связи между этими сущностями с помощью первичных и внешних ключей, обеспечивая целостность и непротиворечивость данных.
>
> В-третьих, MySQL поддерживает механизм транзакций (InnoDB) и блокировки на уровне строк, что важно при параллельной работе нескольких бухгалтеров с одними справочниками и массивами выписок. Это исключает ситуации потери или дублирования данных при одновременной записи, что критично для финансовой системы.
>
> В-четвёртых, СУБД MySQL предоставляет встроенные средства резервного копирования (mysqldump, бинарные логи, репликация) и восстановления, что соответствует требованиям к надёжности и сохранности финансовой информации. Возможность настройки регулярного автоматического резервного копирования базы данных снижает риск утраты данных при сбоях оборудования или программных ошибках.
>
> В-пятых, MySQL распространяется бесплатно в рамках редакции Community Edition, что снижает совокупную стоимость владения системой для небольшой организации и позволяет избежать лицензионных затрат на дополнительное программное обеспечение.
>
> Наконец, в проекте MySQL выступает в роли промежуточного хранилища обработанных банковских выписок перед передачей их в учётную систему 1С:Предприятие. Веб-приложение на PHP выполняет первичную загрузку, распознавание, классификацию и сопоставление операций, после чего структурированные данные сохраняются в MySQL. Оттуда они передаются в 1С через REST API или файловый обмен, при этом типовая конфигурация 1С остаётся неизменной. Такой подход позволяет разделить функции оперативной обработки (PHP/MySQL) и бухгалтерского учёта (1С), упростить интеграцию и снизить нагрузку на учётную систему.

**Схемы:** приведены структурные схемы пользовательской и административной частей, блок-схема обработки платежа, схема управления справочниками [диплом, с. 63–65, рис. 19–24].

Это особенно важно для финансовой системы, где каждое изменение статуса платежа, ручная корректировка контрагента или формирование отчёта должны быть зафиксированы с указанием автора и времени. В MySQL такое журналирование реализуется через таблицу `logs` с внешним ключом к таблице `users`, что обеспечивает связность записей и возможность построения отчётов по действиям персонала.

Поддержка таких типов данных, а также ограничений `NOT NULL`, `UNIQUE` и внешних ключей делает MySQL подходящим выбором для обеспечения точности и достоверности учётных данных.

## 3. Механизм интеграции с 1С

**Место вставки:** подраздел 2.2, после этапа 6 (с. 55–56).

**Текст для вставки:**

> Передача данных в учётную систему 1С:Предприятие 8.3 осуществляется через файловый обмен в формате XML. Обработка загружает файл на стороне 1С и создаёт документы «Поступление на расчётный счёт» или «Списание с расчётного счёта» с заполнением следующих реквизитов: организация, контрагент, договор, сумма, статья движения денежных средств, назначение платежа. В случае невозможности автоматического создания документа (отсутствует контрагент, не заполнен договор, не распознана статья ДДС) система формирует запись в журнале ошибок и направляет уведомление ответственному бухгалтеру. Обмен выполняется по расписанию не реже одного раза в час или по инициативе пользователя. Все операции журналируются для последующего аудита.
>
> **[Почему для промежуточного хранения используется MySQL]** В описанной схеме интеграции MySQL выполняет функцию оперативного промежуточного хранилища (staging area) между источником данных (файлы выписок из интернет-банка) и целевой учётной системой 1С. Такая роль обусловлена тем, что непосредственная загрузка «сырых» выписок в 1С требует доработки типовой конфигурации, что усложняет внедрение и повышает риски ошибок при обновлениях 1С. Веб-приложение на PHP, работающее с MySQL, берёт на себя весь этап предварительной обработки: парсинг форматов 1CClientBankExchange, TXT, CSV, XML, распознавание неструктурированных назначений платежа, сопоставление с существующими справочниками и выявление дублей.
>
> После обработки данные сохраняются в MySQL в виде набора нормализованных записей (таблицы `payments`, `clients`, `banks`, `transactions`), из которых формируются файлы обмена для 1С. MySQL обеспечивает удобный доступ к этим записям, позволяет повторно формировать пакеты для 1С при сбоях передачи, вести историю загрузок и статусы обработки каждой операции. Благодаря этому интеграция становится управляемой: можно отследить, какие именно операции переданы в 1С, какие ожидают повторной отправки, а какие требуют ручной корректировки.
>
> Использование MySQL также позволяет реализовать асинхронный обмен с 1С: веб-приложение записывает подготовленные данные в базу, а фоновый процесс по расписанию формирует и передаёт пакеты в 1С. Это развязывает пользовательский интерфейс и учётную систему по времени: бухгалтер может загружать выписки в любой момент, а обмен с 1С выполняется независимо, например раз в час или по запросу. Все статусы и ошибки обмена фиксируются в базе данных, что обеспечивает прозрачность и возможность аудита.

После подтверждения статус меняется на «активен», и контрагент становится доступен для автоматического сопоставления в последующих выписках. Аналогично организуется работа с нераспознанными назначениями платежа: бухгалтер классифицирует операцию, и созданное правило сохраняется в таблице шаблонов для применения к будущим выпискам.
>
> Журналирование в MySQL также позволяет формировать аналитические отчёты по исключениям: сколько операций требовало ручной обработки за период, какие контрагенты чаще всего вызывали ошибки, каков процент автоматического разнесения. ## 8. Резервное копирование и восстановление

**Место вставки:** подраздел 2.1, раздел «Риски» (с. 48–50).

**Текст для вставки:**

> Для снижения риска потери данных предусмотрен следующий регламент резервного копирования:
> - резервное копирование базы данных — ежедневно в 02:00;
> - хранение копий — локально на сервере организации и в облачном хранилище;
> - срок хранения резервных копий — 30 дней;
> - тестовое восстановление из резервной копии — не реже одного раза в квартал;
> - процедура восстановления после сбоя включает: остановку сервиса, восстановление последней актуальной копии, проверку целостности данных, запуск сервиса и уведомление пользователей.
>
> **[Резервное копирование MySQL]** Поскольку все критичные данные системы (распознанные выписки, справочники, журналы действий, статусы обмена с 1С) хранятся в базе данных MySQL, резервное копирование базы является центральным элементом обеспечения отказоустойчивости. MySQL предоставляет несколько встроенных механизмов создания резервных копий, которые могут использоваться в проекте:
>
> - **Логическое резервное копирование (mysqldump)** — создаёт SQL-дамп базы данных, содержащий структуру таблиц и данные. Этот способ прост в настройке и удобен для периодического полного резервного копирования, например ежедневно в 02:00.
> - **Бинарные логи (binary log)** — фиксируют все изменения данных после создания последнего полного дампа. Их использование позволяет восстановить базу до произвольного момента времени (point-in-time recovery), что снижает потери данных при сбоях.
> - **Репликация MySQL** — позволяет поддерживать резервную копию базы данных на отдельном сервере в режиме, близком к реальному времени. В случае выхода из строя основного сервера можно переключиться на реплику.
>
> Хранение копий как локально, так и в облачном хранилище соответствует требованиям по защите данных от локальных сбоев и инцидентов информационной безопасности. Регулярное тестовое восстановление (не реже одного раза в квартал) позволяет убедиться в работоспособности резервных копий и в корректности процедуры восстановления. После восстановления базы данных выполняется проверка целостности данных: сверка количества записей, контрольные суммы критичных таблиц, проверка работоспособности внешних ключей и индексов.

## 9. Согласование технологического стека

**Место вставки:** подраздел 1.4 (с. 36–41) и 2.4 (с. 60–62).

**Текст для вставки (вариант с веб-приложением + интеграция с 1С):**

Веб-приложение выполняет функции загрузки, распознавания и первичной обработки выписок, а также формирования управленческой отчётности. Данная архитектура позволяет сохранить типовую конфигурацию 1С неизменной и обеспечить быстрый доступ пользователей через браузер.
>
> **[Для чего в этой архитектуре нужна MySQL]** В представленной архитектуре СУБД MySQL выполняет несколько важных функций и является связующим звеном между пользовательским веб-интерфейсом, механизмами обработки выписок и учётной системой 1С.
>
> **1. Хранилище первичных и обработанных данных.** Все банковские выписки, загруженные пользователями, после парсинга и распознавания сохраняются в таблицах MySQL. Это позволяет отделить этап сбора и первичной обработки информации от этапа бухгалтерского учёта в 1С. Вместо того чтобы напрямую создавать документы в 1С из «сырых» файлов, система сначала структурирует данные в MySQL: нормализует реквизиты контрагентов, классифицирует операции, проверяет на дубли и только после этого передаёт готовый пакет в 1С. Это снижает риск создания некорректных документов в учётной системе.
>
> **2. Оперативный доступ через веб-интерфейс.** Поскольку приложение реализовано как веб-приложение на PHP, MySQL является естественным выбором для хранения состояния приложения. Пользователи авторизуются в системе, загружают файлы, просматривают списки операций, выполняют ручное сопоставление — все эти действия требуют быстрого чтения и записи данных. MySQL обеспечивает низкое время отклика для типовых операций веб-приложения и поддерживает одновременную работу нескольких пользователей.
>
> **3. Поддержка справочников и классификаторов.** В MySQL хранятся справочники, необходимые для автоматической обработки выписок: контрагенты, банки, договоры, статьи движения денежных средств, типы платежей, шаблоны распознавания назначений платежа. Индексы по ключевым полям (ИНН, КПП, БИК, расчётный счёт) позволяют мгновенно находить нужного контрагента или банк при сопоставлении очередной операции. Это критично для скорости обработки больших массивов выписок.
>
> **4. Журналирование и аудит.** MySQL используется для хранения журналов действий пользователей, статусов обработки операций, истории обмена с 1С и журнала ошибок. Это позволяет ответить на вопросы: кто загрузил выписку, когда была выполнена обработка, какие операции переданы в 1С, какие требуют ручной проверки. Для финансовой системы такая прослеживаемость обязательна.
>
> **5. Гибкость интеграции с 1С.** Использование MySQL в качестве промежуточного хранилища делает интеграцию с 1С управляемой. Веб-приложение может передавать данные в 1С через REST API (формат JSON) или через файловый обмен, формируя пакеты из записей, уже проверенных и сохранённых в базе. Если обмен с 1С временно недоступен, данные не теряются: они остаются в MySQL со статусом «ожидает передачи», и повторная отправка выполняется автоматически после восстановления связи. Также статусы обработки, возвращаемые 1С, сохраняются в MySQL для дальнейшего анализа.
>
> **6. Экономическая целесообразность.** MySQL Community Edition распространяется бесплатно, что важно для небольшой организации. При этом она обеспечивает достаточный уровень производительности, надёжности и безопасности для задачи обработки банковских выписок. Совместное использование PHP и MySQL снижает стоимость лицензий, упрощает подбор персонала для сопровождения и позволяет развёртывать систему на стандартном серверном оборудовании.
>
> Таким образом, MySQL в данном проекте нужна не как самостоятельная учётная система, а как надёжное, производительное и экономически обоснованное хранилище данных, обеспечивающее корректную работу веб-приложения, промежуточную обработку выписок и управляемый обмен с 1С:Предприятие.
